# Git 基礎知識必備指南

本指南將為您解說 Git 版本控制系統中最核心的四大疑問：**Git 與 GitHub 有什麼差別？**、**為什麼要用 Git？**、**為什麼要有暫存區？** 以及 **Commit 提交訊息該怎麼寫？**

---

## 1. Git 與 GitHub 有什麼差別？

簡單來說：**「Git 是一個工具（軟體），而 GitHub 是一個網站（平台）。」**

我們可以用以下的生動比喻來快速理解：

* **Git 就像是「照相機」**：是一個安裝在您電腦上的工具，用來幫您的檔案拍照、記錄版本歷史。完全不需要網路也可以在本地正常使用。
* **GitHub 就像是「雲端相簿（如 Google 相簿或 Instagram）」**：是一個雲端網站。您可以把用 Git（照相機）拍好的照片（版本紀錄）上傳到 GitHub（雲端相簿）備份，或是分享給別人看、與他人協作。

### 📊 Git 與 GitHub 快速比較表

| 比較項目 | Git | GitHub |
| :--- | :--- | :--- |
| **本質** | 版本控制**工具（軟體）** | 託管 Git 儲存庫的**雲端平台（網站）** |
| **安裝位置** | 安裝在您的**本地電腦** | 運行於**雲端伺服器** |
| **網路需求** | **不需要**網路（可完全離線工作） | **需要**網路（用來上傳、下載與線上操作） |
| **主要功能** | 紀錄檔案修改歷史、建立與合併分支。 | 備份程式碼、團隊協作、專案管理、Pull Request 審查。 |
| **圖形介面** | 主要透過命令列（CLI），或搭配第三方 GUI 軟體。 | 提供美觀的網頁視覺化介面（Web UI）。 |

### 🔗 Git 與 GitHub 的關係是什麼？

他們兩者的關係是**「核心工具」**與**「雲端平台」**的合作模式：

1. **GitHub 建立在 Git 之上（依賴關係）**：
   GitHub 的所有核心功能都是圍繞著 Git 開發的。沒有 Git 的版本控制資料，GitHub 就只是一個普通的雲端硬碟（例如 Google Drive 或 Dropbox）。因為有了 Git，GitHub 才能在網頁上幫您畫出漂亮的分支圖、對比檔案修改差異（Diff），並進行團隊程式碼審查（Pull Request）。
2. **Git 不需要 GitHub 也能單獨使用（獨立性）**：
   Git 是一個完全獨立的單機軟體。即使您不上網、不註冊 GitHub 帳號，您仍然可以在自己電腦上利用 Git 進行完美的版本歷史紀錄。您也可以選擇將 Git 紀錄推送到其他類似平台（例如 GitLab、Bitbucket 或自建伺服器）。
3. **日常的合作流程**：
   當您在電腦上用 **Git** 做好筆記的存檔（Commit）後，透過網路指令（`git push`）將這些紀錄發送到 **GitHub**。GitHub 接收後會解析這些 Git 資料，並將檔案用好看的網頁畫面呈現給您，方便您閱讀或分享。

### 💡 應用場景：什麼時候用 Git、GitHub、一起用，或都不用？

在實際的工作或學習中，您可以根據不同的需求，彈性選擇最適合的方案：

#### 1. 什麼情況下「只用 Git」？（本地控制，不上雲端）

* **應用場景**：
  * **純個人開發/寫作**：不需要與他人協作，也不需要將檔案上傳到網路上。
  * **高度機密專案**：檔案包含商業機密或個人隱私，絕對不能上傳到任何雲端平台。
  * **無網路環境**：在沒有網路的環境（如飛機上、深山）工作，但仍需要記錄版本歷史。
* **原因**：Git 在本地電腦就能完美運作，即使沒有 GitHub 帳號，也能享受版本還原與分支管理的好處。

#### 2. 什麼情況下「只用 GitHub」？（直接網頁操作）

* **應用場景**：
  * **臨時修改文件**：只想快速修正 README 中的一個錯字，可以直接用 GitHub 的網頁版瀏覽器編輯器修改並 Commit。
  * **閱讀與下載資源**：只是想瀏覽別人的開源專案、下載程式碼包（ZIP），或是提交 Issue（回報問題）。
  * **輕量筆記本**：純粹把 GitHub 當作一個線上的 Markdown 筆記閱讀器。
* **原因**：不需要在本地電腦安裝 Git 軟體，直接透過網頁就能完成簡單的編輯與社群互動。

#### 3. 什麼情況下「兩者一起用（Git + GitHub）」？【最推薦、最常見】

* **應用場景**：
  * **團隊協作開發**：多人共同開發一個網站、App 或整理共享知識庫。需要有人寫程式、有人審查代碼。
  * **多裝置同步與備份**：自己在家用桌電寫筆記，出門用筆電寫，需要透過 GitHub 進行進度同步，同時作為安全備份。
  * **建立個人作品集**：將自己在本地用 Git 寫好的專案推送到 GitHub 公開展示，累積個人的學習軌跡。
* **原因**：結合了 Git 的「本地強大控制力（離線、速度快、功能完整）」與 GitHub 的「雲端協作與分享能力」，這是目前軟體開發的標準流程。

#### 4. 什麼情況下「都不用」？

* **應用場景**：
  * **一次性臨時檔案**：隨手記下的雜事清單、用完即丟的草稿。
  * **使用具備自動存檔/共同編輯的軟體**：例如使用 Google Docs、Notion、Notepad 或 Word 寫文件。這些軟體本身就內建了自動雲端備份或歷史紀錄功能。
* **原因**：導入 Git 會增加額外的指令學習成本，對於不需要版本歷史比對的單純文件，使用一般的文書處理工具更直觀。

## 2. 為什麼要用 Git？

在沒有使用 Git 之前，我們備份檔案或專案通常是這樣做的：

* `study-notes_2026-07-10.zip`
* `study-notes_最新版.zip`
* `study-notes_絕對最終版.zip`
* `study-notes_絕對最終版_修改版.zip`

這種方式會帶來嚴重的痛苦：不知道哪個才是真正的最新版、無法得知每次具體改了什麼，而且極佔硬碟空間。

### Git 帶來的四大核心價值

1. **版本時光機**：
   Git 會記錄您每一次的「存檔點（Commit）」。不論您是不小心刪除了寫好的 Markdown 筆記，還是修改程式碼後導致系統崩潰，您都可以隨時一鍵還原到過去的任何一個時間點。
2. **精確的修改紀錄（Diff）**：
   Git 能精確顯示每一行文字的「新增」、「刪除」與「修改」。您可以隨時查閱是誰、在什麼時候、因為什麼原因改了這行內容。
3. **分支管理（Branching）**：
   您可以建立一個獨立的分支（Branch）去嘗試新想法或寫草稿，而不會影響到原本乾淨的「主要分支（main）」。等確定內容完美無誤後，再合併（Merge）回來。
4. **雲端備份與多人協作**：
   搭配 GitHub，您的檔案能安全地備份在雲端。多人可以同時編輯同一個專案，Git 會自動處理合併，若有衝突也會明確標示出來，防止互相覆蓋檔案。

## 3. 為什麼要有暫存區（Staging Area）？

在 Git 的設計中，從「修改檔案」到「正式存檔」，中間多了一個叫**暫存區（Staging Area）**的緩衝地帶。

很多剛接觸 Git 的人會問：*「為什麼不直接讓我把修改好的檔案直接 Commit 存檔，還要先執行 `git add` 放到暫存區，這不是很繁瑣嗎？」*

我們可以把這個流程想像成**「網路購物」**：

```text
 🛒 賣場商品 (工作區)  ──[ 放進購物車 (git add) ]──>  🛒 購物車 (暫存區)  ──[ 結帳付款 (git commit) ]──>  📦 收到包裹 (本地儲存庫)
```

### 暫存區不可或缺的三大理由

1. **精準打包，保持提交歷史乾淨**：
   假設您今天在筆記庫裡同時做了三件事：
   * 修改了 `JavaScript_筆記.md` 的錯字（已完成）
   * 新增了 `原子習慣_讀書心得.md`（已完成）
   * 寫了一半的 `Python_基礎.md`（尚未寫完）

   如果沒有暫存區，您一執行 Commit 就會把「寫了一半、還不能看」的 Python 筆記一起存檔。
   有了暫存區，您可以只挑選已完成的檔案放進「購物車」：

   ```bash
   git add JavaScript_筆記.md
   git add 原子習慣_讀書心得.md
   ```

   然後結帳：

   ```bash
   git commit -m "docs: 新增原子習慣讀書心得與修正 JS 錯字"
   ```

   那份寫到一半的 `Python_基礎.md` 則會安全地留在您的電腦（工作區）中，不會被一起打包。這能讓每一次的 Commit 主題都非常單一且明確。

2. **提供最後確認的機會**：
   在檔案被放進暫存區後、正式結帳（Commit）前，您可以透過 `git diff --cached` 仔細檢查到底有哪些變更即將被寫入歷史。這能有效避免把敏感資訊（如個人密碼、金鑰）或垃圾暫存檔不小心存檔。

3. **分批提交**：
   如果您一口氣修改了 10 個檔案，您可以利用暫存區將它們分門別類，拆編成 3 個意義不同的 Commit 來提交，而不是塞成一個巨大且混亂的 Commit。

## 4. git commit -m "提交訊息" 裡的提交訊息要寫什麼？

Commit 訊息（Commit Message）是寫給**「未來的自己」**以及**「未來的合作夥伴」**看的說明書。

如果您的提交訊息總是寫：`update`、`test`、`fix`、`111`，那麼當幾個月後專案出現問題，您想要尋找特定歷史版本時，您會完全無法從紀錄中分辨每個版本在幹嘛。

### ✍️ 好的 Commit 訊息撰寫原則

* **簡短且具描述性**：用一句話（建議 50 字以內）說明這次變更的「目的」，而不是變更的「細節」（細節看程式碼差異即可）。
* **不要寫廢話**：例如「修改了檔案」是廢話，因為 Git 本來就知道你修改了檔案。請寫「為什麼修改」或「修改了什麼內容」。

### 🚀 業界主流規範：約定式提交 (Conventional Commits)

推薦採用目前全球開發者通用的格式：**`型態(選填範圍): 簡短說明`**

常見的型態（Type）有以下幾種：

| 型態 (Type) | 適用情境 | 範例 |
| :--- | :--- | :--- |
| **`feat`** (feature) | 新增功能、新增筆記或新檔案 | `feat: 新增原子習慣讀書心得` |
| **`fix`** | 修復 Bug、修正筆記錯字 | `fix: 修正 JavaScript 閉包說明中的語法錯誤` |
| **`docs`** (documentation) | 單純修改文件、README、目錄結構說明 | `docs: 更新總目錄超連結` |
| **`style`** | 不影響內容的格式調整（如排版、空白鍵、換行） | `style: 調整排版與標題層級` |
| **`refactor`** | 重構（非新增功能也非修 Bug，例如重整資料夾結構） | `refactor: 重整程式開發分類下的子目錄` |
| **`chore`** | 其他雜事（例如修改設定檔、gitignore 等） | `chore: 新增 gitignore 忽略 Obsidian 暫存檔` |

### 🙅 壞範例 vs 🙆 好範例

* ❌ **壞範例**：`update`
  🙆 **好範例**：`docs: 更新手機晶片大廠比較表`
* ❌ **壞範例**：`修正錯字` （不知道修正了哪一檔的錯字）
  🙆 **好範例**：`fix: 修正 0710御三家.md 中的專有名詞錯字`
* ❌ **壞範例**：`寫了一堆程式碼，順便整理了目錄，然後修了一個 bug` （主題太雜，應該拆開 Commit）
  🙆 **好範例**：拆分成兩個 Commit：
  1. `feat: 新增 OpenAI API 串接功能`
  2. `style: 調整專案目錄結構`
